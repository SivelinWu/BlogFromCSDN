#### 一、水平权限漏洞：定义与原理
​**​水平权限（Horizontal Privilege Escalation）​** ​ 指​**​同一权限等级的用户非法访问或操作其他用户资源​** ​的安全问题。例如：用户A通过篡改请求参数，越权访问用户B的订单、地址或私有数据。其核心原因是：​**​服务端未校验请求者与资源的归属关系​** ​，典型漏洞模式包括：
  1. ​**​缺失资源归属校验​** ​：接口仅根据ID操作资源（如`/getOrder?orderId=1001`），未验证当前用户是否属于该资源所有者。
  2. ​**​可预测的资源标识符​** ​：使用自增数字ID（如1001、1002），攻击者通过遍历ID即可获取他人数据。
  3. ​**​过度信任客户端参数​** ​：直接使用客户端传入的ID执行数据库操作，未结合会话（Session）或令牌（Token）中的用户身份校验。


* * *
#### 二、典型案例与风险场景
##### 案例1：用户信息越权查询
  * ​**​漏洞接口​** ​：`GET /user/profile?userId=123`
  * ​**​攻击方式​** ​：  
用户A（自身ID=123）登录后，修改`userId=124`，成功获取用户B的隐私信息（手机号、地址等）。
  * ​**​根本原因​** ​：  
服务端仅根据`userId`查询数据，未校验当前会话用户ID与目标`userId`是否匹配。


##### 案例2：订单越权删除
  * ​**​漏洞接口​** ​：`DELETE /order?orderId=2001`
  * ​**​攻击方式​** ​：  
用户A将`orderId`改为用户B的订单ID，触发删除操作。
  * ​**​根本原因​** ​：  
删除前未验证订单所属用户，SQL语句为： 
        DELETE FROM orders WHERE order_id = 2001; // 缺失AND user_id = ?
**风险影响：**
  * **​数据泄露​** ​：获取他人敏感信息（如银行卡、聊天记录）。
  * ​**​数据篡改​** ​：恶意修改他人资料、订单状态或支付信息。
  * **​法律风险​** ​：违反GDPR等隐私法规，导致企业面临高额罚款。

#### 三、测试的识别与定位
##### ​**​1. 漏洞特征识别与测试场景设计​** ​
水平权限漏洞的核心特征是：​**​接口未校验资源归属关系​** ​，攻击者可篡改参数访问同级用户数据。测试需重点关注以下场景：
  * ​**​关键参数类型​** ​： 
    * ​**​直接标识符​** ​：`user_id`、`order_id`、`mobile` 等（如 `GET /user/info?user_id=1001`）
    * ​**​间接标识符​** ​：哈希值、UUID（若未绑定用户会话仍可被遍历）
  * ​**​高危操作接口​** ​： 
    * 数据查询类：用户资料、订单详情、私密内容访问
    * 数据修改类：个人信息更新、密码重置、状态变更
    * 数据删除类：订单取消、资源删除


> 📌 ​**​测试用例设计模板​** ​：
> `用例ID：HORIZONTAL-001   
>  测试目标：验证订单详情接口是否存在水平越权   
>  步骤：   
>  1. 用户A登录，访问自身订单详情（order_id=1001），记录响应数据   
>  2. 用户B登录，访问同一接口但替换order_id=1001   
>  3. 对比两次响应：若B成功获取A的订单数据→漏洞存在 `
##### **​2. 漏洞识别**
  * ​**​工具链​** ​：Charles、Postman、浏览器开发者工具
  * ​**​操作流程​** ​： 
    1. ​**​账号准备​** ​：至少两个同权限测试账号（A/B）
    2. ​**​请求捕获​** ​： 
       * A账号操作目标接口（如查看订单），Charles抓包记录请求`R1`
       * 提取关键参数：如 `{"order_id": 1001, "token": "A_token"}`
    3. ​**​参数篡改重放​** ​： 
       * ​**​方式1​** ​：保留A的`token`，替换`order_id`为B的资源ID
       * ​**​方式2​** ​：保留B的`order_id`，替换`token`为A的凭证
    4. ​**​结果验证​** ​： 
       * 响应码200且返回数据完整 → 漏洞存在
       * 对比A/B账号响应数据一致性（自动化脚本推荐）

#### 四、解决方案
##### 1\. ​**​强制服务端权限校验（推荐）​** ​
在业务逻辑层显式校验资源归属：
    
    // 伪代码示例：订单查询接口  
    public Order getOrder(Long orderId) {  
        Long currentUserId = getCurrentUserId(); // 从Session或Token获取当前用户ID  
        Order order = orderDao.findById(orderId);  
        if (order == null || !order.getUserId().equals(currentUserId)) {  
            throw new PermissionDeniedException("无权访问此订单");  
        }  
        return order;  
    }  
​**​优点​** ​：实现简单，逻辑清晰；
​**​缺点​** ​：增加一次数据库查询。
##### 2\. ​**​SQL层强制归属约束​** ​
在数据访问层嵌入用户ID条件，确保查询与当前用户绑定：
    
    SELECT * FROM orders WHERE order_id = #{orderId} AND user_id = #{currentUserId};
​**​适用场景​** ​：资源表已包含用户ID字段（如`user_id`）。
##### 3\. ​**​使用不可预测的资源标识符​** ​
  * 将自增ID替换为​**​UUID​** ​或​**​随机哈希值​** ​（如`/order/detail?id=abcd1234efgh`）。
  * ​**​局限性​** ​：仍需结合归属校验（避免爬虫或链接泄露导致越权）。


##### 4\. ​**​Token二次验证（防参数篡改）​** ​
  * **生成Token​** ​： 
        String token = md5(resourceId + sessionId + "私钥");
​**​接口验证​** ​：
  * 请求时同时传递`resourceId`和`token`，服务端重新计算并校验一致性。


##### 5\. ​**​安全框架统一管控（Spring Security/Shiro）​** ​
使用安全框架在拦截层统一校验权限，避免代码重复：
    
    // Spring Security示例：基于注解的权限控制  
    @PreAuthorize("@orderService.isOwner(#orderId, authentication.principal.id)")  
    @GetMapping("/order/{orderId}")  
    public Order getOrder(@PathVariable Long orderId) {  
        // 直接执行业务逻辑  
    }  
在`OrderService`中实现归属判断方法`isOwner()`。
* * *
#### 五、总结​
  1. ​**​设计阶段​** ​： 
     * 强制要求所有资源操作接口声明资源归属字段（如`@OwnerField("user_id")`）
  2. ​**​测试阶段​** ​： 
     * 接口测试覆盖100%含ID参数的读写操作
     * 自动化注入越权检测到流水线（如Burp+Jenkins联动）
  3. ​**​监控阶段​** ​： 
     * 日志告警规则：单账号高频访问不同用户资源（如1分钟访问50个不同user_id）


通过以上流程，测试团队可系统性识别水平权限漏洞，推动研发从​**​被动修复​** ​转向​**​主动预防​** ​，构建权限校验的闭环防御体系。
