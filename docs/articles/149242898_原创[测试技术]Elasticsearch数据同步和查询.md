## 一、ES（elasticsearch）介绍
ES（elasticsearch）是个开源分布式搜索引擎，提供搜集、分析、存储数据三大功能。
在实际项目中经常将 MySQL 作为业务数据库，ES 作为查询数据库，用来实现读写分离，缓解 MySQL 数据库的查询压力，应对海量数据的复杂查询。
## 二、ES同步—实现方式
##### **2.1、同步双写：强实时性但高耦合​** ​
​**​实现原理​** ​  
在业务代码中同步写入MySQL与ES，通过本地事务保证基础一致性。
**✅ 核心优势​** ​
  * ​**​实时性高​** ​：数据写入后立即可查（延迟<100ms）
  * ​**​架构简单​** ​：无需额外中间件，适合小型项目快速落地


​**​⚠️ 致命缺陷​** ​
  * ​**​业务侵入性强​** ​：需在所有写入逻辑中硬编码ES操作，维护成本高
  * ​**​分布式事务风险​** ​：MySQL成功但ES失败时需手动回滚（建议引入本地事务表+重试机制）
  * ​**​性能瓶颈​** ​：双写使TPS下降30%+，高并发场景易触发线程阻塞


​**​🔥 避坑指南​** ​
  * ​**​补偿机制​** ​：记录失败操作到本地表，定时任务重试
  * ​**​批量化写入​** ​：合并多次写入为ES Bulk请求，降低网络开销


##### **2.2、异步双写：解耦架构的折中选择​** ​
​**​实现原理​** ​  
MySQL写入后发送MQ消息，消费者异步更新ES，实现业务与搜索解耦。  
_原理示意：MySQL → Kafka → ES Consumer_
​**​✅ 核心优势​** ​
  * ​**​吞吐量提升​** ​：MQ削峰填谷，支撑万级QPS
  * ​**​故障隔离​** ​：ES宕机不影响主业务，消息可堆积重试


​**​⚠️ 关键挑战​** ​
  * ​**​消息顺序性​** ​：同一数据的多次更新需保证顺序（通过分区键绑定同一分区）
  * ​**​消费延迟​** ​：突发流量下消息堆积导致延迟（需监控Lag值）
  * ​**​数据漂移​** ​：极端情况下可能出现短时数据不一致


##### **2.3、定时同步：离线场景的保守策略​** ​
​**​实现原理​** ​  
定时扫描MySQL增量数据（如每10分钟），通过Logstash批量同步到ES。
**✅ 适用场景​** ​
  * 允许小时级延迟的报表分析
  * 历史数据迁移（TB级离线同步）


​**​⚠️ 硬伤限制​** ​
  * ​**​时效性差​** ​：默认配置下延迟≥10分钟
  * ​**​分页性能差​** ​：深分页查询可能触发OOM（需游标分页优化）


​**​🚀 性能提升技巧​** ​
  * ​**​断点续传​** ​：持久化`sql_last_value`到Redis防宕机中断
  * ​**​冷热分离​** ​：仅同步近7天热数据，历史数据通过ES冷节点存储


##### **2.4、Binlog监听：生产级实时同步方案​** ​
​**​实现原理​** ​  
通过Canal/Debezium订阅MySQL Binlog，实时推送变更事件到ES。  
_原理示意：MySQL → Canal → Kafka → ES Worker_
​**​✅ 核心优势​** ​
  * ​**​零业务侵入​** ​：完全解耦业务代码
  * ​**​秒级延迟​** ​：99%场景下延迟<1秒
  * ​**​精准数据捕获​** ​：支持UPDATE/DELETE操作（Logstash无法原生支持）


​**​⚠️ 运维复杂度​** ​
  * ​**​组件运维成本​** ​：需维护Canal/Kafka集群
  * ​**​消息丢失风险​** ​：网络抖动可能导致事件丢失（需ACK确认+死信队列）


## 三、使用Kibana查询
#### 3.1、**Kibana简介**
Kibana 是为Elasticsearch​设计的开源分析和可视化平台。你可以使用 Kibana 来搜索，查看存储在 Elasticsearch 索引中的数据并与之交互。你可以很容易实现高级的[数据分析](https://cloud.tencent.com/product/bi?from=20065&from_column=20065 "数据分析")数据分析[数据分析](https://cloud.tencent.com/product/bi?from=20065&from_column=20065 "数据分析")和可视化，以图标的形式展现出来。
#### **3.2、Kibana的使用场景**
**1.** 实时监控通过 histogram 面板，配合不同条件的多个 queries 可以对一个事件走很多个维度组合出不同的时间序列走势。时间序列数据是最常见的监控报警了。**2.** 问题分析关于 elk 的用途，可以参照其对应的商业产品 splunk 的场景：使用 Splunk 的意义在于使信息收集和处理智能化。而其操作智能化表现在：**搜索** 通过下钻数据排查问题，通过分析根本原因来解决问题；**实时可见性** 可以将对系统的检测和警报结合在一起，便于跟踪 SLA 和性能问题；**历史分析** 可以从中找出趋势和历史模式，行为基线和阈值，生成一致性报告。
#### 3.3、使用KQL查询
**KQL（Kibana Query Language）** ，也就是在Kibana上面进行查询时使用的语法。
> 在`Kibana`中进行查询的时候，建议使用指定索引查询，这样的效率更高，而不建议使用全局查找的方式。
###### 3.3.1、**字段精准匹配​**
  * ​**​精确值查询**


    
    status: 200                # 匹配 status 字段值为 200 的文档
    http.method: "GET"         # 字符串需用英文双引号包裹
  * ​**​字段是否存在**


    
    response:*                 # 匹配包含 response 字段的文档（无论值为何）
  * **分词匹配​** ​（不指定字段）


    
    error timeout              # 匹配包含 "error" 或 "timeout" 任意一词的文档
  * **短语匹配​** ​（严格顺序）


    
    message: "connection refused"  # 必须完整匹配整个短语
###### 3.3.2、逻辑运算符组合
  * **布尔逻辑**


    
    status: 500 AND service: "payment"     # 同时满足两个条件
    error OR warn                          # 包含 error 或 warn 任意一词
    NOT response: 200                      # 排除 response 为 200 的文档
  * ​**​优先级控制**


    
    (status: 200 OR status: 404) AND latency > 1000  # 括号内 OR 优先执行
    response: 200 AND NOT (extension:php OR extension:css)  # 排除特定扩展名
###### 3.3.3、高阶查询技巧
  * **通配符与模糊匹配​**


    
    host.name: web-server-*       # 匹配以 "web-server-" 开头的主机名
    message: "timeout*"            # 匹配以 "timeout" 开头的消息
    FlightNum: T*V                 # 字段值以 T 开头、V 结尾（如 "T5314V"）
  * **范围查询**


    
    response_time: > 500          # 响应时间大于 500ms
    bytes: 1000..5000             # 字节数在 1000 到 5000 之间（等效于 >=1000 AND <=5000
    @timestamp > "2025-07-10T00:00"  # 指定时间点之后
  * ​**​嵌套字段查询​**


    
    # 匹配单层嵌套（如 items 字段下的 name 为 "banana"）
    items: { name: "banana" }
    
    # 多层嵌套路径（如 level1 下的 level2 字段）
    level1.level2: { prop1: "foo" AND prop2: "bar" }  # 同一嵌套对象需同时满足
  * **日期范围高级用法**


    
    @timestamp > now()-1h          # 最近 1 小时内的数据
    @timestamp between "2025-07-01" and "2025-07-10"  # 跨日期区间
**注意：**
  1. ​**​大小写敏感性问题​** ​  
KQL ​**​不区分大小写​** ​（如 `ERROR` 和 `error` 等效）。
  2. ​**​通配符限制​** ​ 
     * 禁止前导通配符（如 `*error`）除非修改 Kibana 高级设置。
     * 避免过度使用 `*` 导致性能下降。
  3. ​**​与 Lucene 语法区别​** ​  
KQL ​**​不支持正则表达式​** ​（如 `message: /Error \d+/` 无效）。


掌握以上语法可覆盖 95% 的日常分析场景，建议保存常用查询为 Kibana ​**​Saved Search​** ​ 提升团队效率。
