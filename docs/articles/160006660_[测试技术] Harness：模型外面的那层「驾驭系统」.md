# [测试技术] Harness：模型外面的那层「驾驭系统」

> 原文: https://blog.csdn.net/weixin_42390585/article/details/160006660

> *原创内容，未获授权禁止转载、转发、抄袭。

## 
大家聊 Agent 时，注意力常常落在模型牌子上：谁又发新权重了，谁又刷榜了。可最近圈里更常出现的词反而是 Harness。有人把它当成新框架，有人当成热词——mCell 那篇文章里有一句判断我很认同：它不是什么横空出世的「新发明」，而是把大家一直在做的那类软件工程现实，终于叫出了名字。
下面把 Harness 是什么、和 Context 怎么分家、落地时长什么样，一次说清楚。
* * *
### Harness 到底是什么？
先给一句 working definition，糅合几篇文章的共识：
Harness 不是模型本身，也不是单条 Prompt、不是某一个 Tool Call、更不是某个具体框架名字；它是 Agent 运行时的「工程环境总和」——任务怎么表达、上下文怎么装配、工具怎么暴露与拦截、状态怎么存取与裁剪、错误怎么分类与重试、反馈怎么回流成模型能用的信号、安全边界落在哪、以及系统怎么验收「事真的做完了」。
换个比喻：模型像「会调用工具的大脑」，Harness 就是身体、传感器、护栏、操作台、流程和回执——甚至包括出事之后的兜底。mCell 把它译作 「驾驭系统」：重点不在炫技，在能不能被稳定驾驭。
社区里常用的式子可以写成：
Agent = Model + Harness（或 Agent = Loop(Model + Harness)，把 Context 和 Tools 都收进 Harness 里谈；与 OpenAI Codex 实践里 coding agent = AI model(s) + harness 是同一类说法。）
Martin Fowler 在 Harness Engineering 相关讨论里给过一个特别利落的二分：Guides（前馈）——行动之前引导它别走偏；Sensors（反馈）——行动之后帮它看见后果并自我纠正。只有一边，都会瘸腿：光反馈会重复犯同类错；光前馈不知道规则有没有真起作用。
* * *
### 它跟 Prompt、Context 差在哪？
三层递进可以这样记：
层次| 关心什么  
---|---  
Prompt Engineering |  怎么说——措辞、few-shot、角色  
Context Engineering |  给什么——塞哪些文档、怎么裁历史、怎么控 bloat  
Harness Engineering |  在什么环境里做——何时能动手、用哪些工具、要不要审批、会话怎么恢复、危险操作怎么拦、客户端怎么展示状态……  
所以：Context Engineering 是 Harness 的一部分，但 Harness 远大于「把上下文喂满」。 OpenAI 官方也强调过「情境是稀缺资源」——巨长指令会挤掉真正该看的任务与代码；解法往往已经越过「写 prompt」，进入仓库结构、审阅回路、验证手段那一层（mCell 文中有对应解读）。
* * *
### 为什么「编辑工具」成了 Harness 的缩影？
老章那篇把矛盾收束得很具体：Agent 写代码的主循环无非是读—想—改—写回，「写回」这一刀的协议设计，各家差异巨大，且没有银弹：`apply_patch`、`str_replace`、神经网络合并等路线各有硬伤（对模型友好度、字符级复现、成本都不相同）。Hashline 这类方案用短哈希锚定行、冲突可检测，让模型少背整段原文、降低无效重试——本质是在降低 Harness 与模型之间的摩擦系数。这和「模型变聪明」是两条线：你在怪飞行员，可能是起落架设计有问题（文中引用的比喻）。
这也解释了近期的 oh-my-* 生态：社区在给不同 CLI/引擎套更厚的 Harness（多角色流水线、意图网关、模型路由、与 Hashline 同思路的锚定编辑等）。Anthropic 的 Claude Managed Agents 则更像官方提供的托管式 Harness / meta-harness：Brain（推理循环）与 Hands（沙箱执行）、Session（事件日志）解耦，Harness 被设计成可替换层——因为「Harness 编码的是对模型能力的假设，模型一变，假设可能过时」。这和「Harness 要轻、要能演进」并不矛盾。
* * *
### 若你要落地：Harness 常拆成哪几块？
mCell 用 memo 实践拆的五层，很适合当检查清单（此处压缩为可执行心智模型）：
  1. 上下文装配：提示词不是单文件百科全书，而是动态拼装（模板、项目规则、Skills、工具 schema 等分层、分优先级）。
  2. 工具治理：工具要可发现、可校验、可分级、可拦截、可审计——否则只是「裸奔的 function 列表」。
  3. 安全与审批：能写盘、能跑 shell 的 Agent 已是系统参与者；边界必须在运行时 enforce，不能指望「prompt 里请自觉」。
  4. 反馈与状态：把系统内部发生的事翻译成模型能消费的反馈（权限、截断、范围过大、命令被拒等），否则它只看到「失败了」三个字。
  5. 熵管理：规则、文档、会话、记忆会腐烂；持续清理与压缩，让系统回到「对模型友好」的状态——OpenAI 那类百万行 Agent 生成代码的实践里，这也是显式主题。


若只记三个词：约束、反馈、稳定（mCell 的收束）——和 Fowler 的 Guides / Sensors 也能对上号。
* * *
### 「Harness 才是分水岭」这句话要不要信？
我的看法是：对一大半。 在模型差距收窄的阶段，Harness 往往是当下最大的杠杆；但 Harness 不是魔法：诊断类错误、过度工程、指令误解等偏「认知」的问题，仍然会被模型能力上限卡住。
更稳的表述是：模型决定能力上限，Harness 决定你能稳定发挥出多少；现阶段投 Harness 的 ROI 往往极高，但别把它当成替代模型进步的永久拐杖。
mCell 给的落地顺序也值得写进博文：先别只追最强模型，先把环境做成可复用资产；工具先系统化成「系统」；上下文不要写成一坨；安全在运行时；失败要可解释；长任务要有状态与恢复；定期做熵管理。
* * *
### 小结
Harness = 包在模型外的、让 Agent 能在真实约束里持续干活的整套运行与治理系统。 它包含 Context，但不止于 Context；它用 Guides/Sensors 把「事前引导」和「事后纠正」绑在一起；在具体工程里，编辑协议、工具治理、安全、反馈语言、熵管理往往比「再换一个大模型」更先撞到天花板。
