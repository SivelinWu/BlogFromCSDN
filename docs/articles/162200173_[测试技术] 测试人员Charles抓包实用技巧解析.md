# [测试技术] 测试人员Charles抓包实用技巧解析

> 原文: https://blog.csdn.net/weixin_42390585/article/details/162200173

> 原创内容，未获授权禁止转载、转发、抄袭。

Charles 很多人都会用，但不少人只停留在“抓到接口，看下返回值”。
真正排查问题时，Charles 不只是抓包工具。它可以帮测试人员做 HTTPS 解密、过滤无效请求、Mock 异常数据、切换接口环境、模拟弱网、重放请求。用好了，很多前后端扯皮的问题能更快定位。
这篇不讲大而全的配置，只讲测试日常最常用的几类场景。
### 用 Map Local 测异常数据
Map Local 适合模拟接口返回。
    
    右键接口 -> Map Local
    
常见场景：
  * 空列表
  * 字段缺失
  * 超长列表
  * 金额异常
  * 状态异常
  * 错误码返回
  * 文案超长


比如订单列表接口，可以本地准备一个空数据：
    
    {
      "code": 0,
      "message": "success",
      "data": []
    }
    
然后看页面是否有空状态提示，而不是一直转圈或白屏。
Map Local 只是替换返回内容，不适合处理复杂动态逻辑。
### 用 Map Remote 切环境
有些 App 包不方便切环境，或者 H5 页面写死了线上域名，可以用 Map Remote。
    
    Tools -> Map Remote
    
示例：
    
    From: https://api.prod.example.com/*
    To:   https://api.test.example.com/*
    
这样客户端不用重新打包，也能把请求转到测试环境。
但 Map Remote 会改变真实请求去向，不要在真实支付、真实下单、真实退款链路上乱转发。建议只在测试账号、测试数据、明确授权的场景里使用。
### Rewrite 适合批量改请求和响应
    
    Tools -> Rewrite
    
常见场景：
  * 自动加 Header
  * 修改 Authorization
  * 替换响应字段
  * 把正常响应改成异常结构
  * 把接口返回改成空数据
  * 构造错误码、字段缺失、超长文案


如果当前 Charles 版本支持响应状态码改写，也可以用 Rewrite 构造 `500`、`403`。若版本不支持，建议用 Breakpoints、Map Local，或者让测试服务直接返回异常状态。
测试前端容错时，不是把 `500` 改成 `200`，而是把正常接口改成异常情况，看前端是否兜得住。
Rewrite 规则一定要先小范围匹配，不要一上来全局生效。
### Breakpoints 适合临时改一次
如果只是临时改一次请求或响应，用 Breakpoints 更合适。
    
    右键接口 -> Enable Breakpoints
    
适合这些场景：
  * 临时改金额
  * 临时改订单状态
  * 临时改用户权限
  * 临时构造边界值
  * 临时测试前端渲染异常


简单理解：
    
    长期规则用 Rewrite。
    临时修改用 Breakpoints。
    
### 弱网测试别只调网速
    
    Proxy -> Throttle Settings
    
Throttle 可以模拟低带宽、高延迟、弱网场景。
重点看：
  * 页面是否有加载态
  * 接口超时是否提示
  * 请求失败是否重试
  * 按钮是否防重复点击
  * 下单中断后订单状态是否一致
  * 支付中网络异常是否能恢复


弱网测试不是把网速调慢就完事，而是看业务状态会不会乱。
### Repeat 用来验证幂等
选中请求后，可以用：
    
    Repeat
    Repeat Advanced
    
`Repeat` 适合单次重放，`Repeat Advanced` 可以配置次数和并发，适合做轻量并发验证。
比如领取优惠券接口重复请求后，要确认：
  * 用户券包不会多一张
  * 库存不会多扣
  * 领取记录只有一条
  * 接口返回明确提示


`Repeat Advanced` 不是专业压测工具，不能替代 JMeter、k6、Locust。也不要对生产环境的下单、支付、退款、发券接口随便做并发重放。
### 一个真实排查流程
比如测试支付成功页时，页面提示“支付成功”，但用户权益没到账。
可以这样查：
  1. 用 Focus 只看业务域名。
  2. 找到支付接口和权益发放接口。
  3. 看支付接口是否返回成功。
  4. 看订单状态是否变成已支付。
  5. 看权益接口是否调用。
  6. 看权益接口返回码和响应体。
  7. 用 Repeat 重放权益查询接口，确认是否是延迟。
  8. 必要时导出 `.chls` 会话给开发复现。


页面成功只说明前端收到了结果，真正要看的是订单、流水、权益、后台记录是否一致。
### 调试完记得关规则
Charles 里最容易踩的坑，不是不会配置，而是忘记关闭配置。
调试结束后建议检查：
    
    Map Local 是否关闭
    Map Remote 是否关闭
    Rewrite 是否关闭
    Throttle 是否关闭
    Breakpoints 是否还开着
    SSL Proxying 是否只保留必要域名
    
否则后面测其他功能时，可能还在使用旧 Mock、旧环境转发或旧弱网配置。
### 推荐默认配置
日常可以按这个思路配置：
    
    1. 只放行必要设备，不长期使用 0.0.0.0/0
    2. SSL Proxying 只配置业务域名，不长期全局 *
    3. Recording 过滤静态资源和埋点请求
    4. Filter 勾选 Filter uses regex，临时使用 GET|POST|PUT|DELETE|PATCH
    5. 用 Focus 聚焦当前业务域名
    6. Map Local 用来 Mock 返回
    7. Map Remote 用来切环境
    8. Rewrite 用来批量改请求和响应
    9. Breakpoints 用来单次临时修改
    10. Throttle 用来测弱网和超时
    11. Repeat 用来验证幂等和重复提交
    12. 调试完关闭 Map、Rewrite、Throttle、Breakpoints
    
### 总结
Charles 的价值不只是抓包，而是帮测试更快定位问题。
测试人员用 Charles，不能只看接口返回值。
要看请求、响应、状态、数据和业务结果是否一致。
