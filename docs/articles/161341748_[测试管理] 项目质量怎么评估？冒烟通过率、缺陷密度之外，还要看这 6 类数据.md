# [测试管理] 项目质量怎么评估？冒烟通过率、缺陷密度之外，还要看这 6 类数据

> 原文: https://blog.csdn.net/weixin_42390585/article/details/161341748

> 原创内容，未获授权禁止转载、转发、抄袭。

项目上线前，最常听到一句话：
    
    测试通过率 98%，可以发版。
    
但线上真出问题时，大家才发现：**通过率高，不代表风险已经消失。**
项目质量评定不是为了给项目打一个漂亮分数，而是回答三个问题：
    
    1. 当前版本能不能继续测？
    2. 当前版本能不能上线？
    3. 上线后最可能出问题的地方在哪里？
    
所以质量不能只看一个指标，要组合看。

### 冒烟通过率：判断能不能继续测
冒烟测试看的是版本准入，不是最终质量。
    
    冒烟通过率 = 冒烟通过用例数 / 冒烟总用例数
    
但冒烟不能只看百分比，要看失败用例的位置。
比如同样是 98%：
    
    失败的是帮助中心入口：可以记录风险后继续测
    失败的是登录、下单、支付：不建议继续系统测试
    
我更建议把冒烟用例分级：

等级| 含义| 处理方式  
---|---|---  
S1| 核心入口、主流程| 失败则阻断测试  
S2| 重要功能| 失败需评估影响  
S3| 低频入口| 可记录风险后继续  

结论：**冒烟通过率看的是“版本有没有继续测试的资格”。**
### 缺陷密度：判断问题集中在哪
缺陷密度常见口径：
    
    缺陷密度 = 缺陷数 / 需求数
    
也可以按功能点、Story Point、代码行数统计。关键是团队内部口径要统一。
示例：

模块| 需求数| 缺陷数| 缺陷密度  
---|---|---|---  
登录| 3| 2| 0.67  
订单| 8| 18| 2.25  
优惠券| 4| 15| 3.75  

这张表说明：风险重点不在登录，而在订单和优惠券。
但缺陷密度不能单独判断质量。测得越深，缺陷可能越多；测得越浅，缺陷反而少。
所以它更适合回答：
    
    问题集中在哪些模块？
    哪些模块需要重点回归？
    哪些需求质量风险最高？
    
### 缺陷级别：判断能不能上线
缺陷数量不是重点，严重程度才是。

级别| 含义| 上线影响  
---|---|---  
P0| 系统不可用、核心流程阻断、数据严重错误| 必须阻断  
P1| 核心功能异常，有明显业务影响| 原则上阻断  
P2| 一般功能问题，有替代方案| 评估后决定  
P3| 文案、样式、轻微体验问题| 可延期  

上线前重点看：
    
    P0 是否清零
    P1 是否清零或有明确兜底方案
    P2 是否集中在核心模块
    延期缺陷是否经过业务确认
    已修复缺陷是否完成回归
    
如果还有支付、权限、数据一致性相关 P1，测试通过率再高也不建议上线。
### 修复趋势：判断质量是否收敛
缺陷修复率公式：
    
    缺陷修复率 = 已关闭缺陷数 / 缺陷总数
    
但只看修复率不够。项目后期更要看趋势。

阶段| 健康趋势| 风险信号  
---|---|---  
测试初期| 缺陷快速暴露| 缺陷很少，可能没测透  
测试中期| 修复量追上新增量| 重开率高、修复慢  
测试后期| 新增缺陷明显下降| 严重缺陷仍在新增  

临上线前，如果新增缺陷还高于修复缺陷，说明版本还没收敛。
### 重开率：判断修复质量
重开率经常被忽略，但很能说明问题。
    
    缺陷重开率 = 重开缺陷数 / 已修复缺陷数
    
重开率高，通常说明：
    
    开发只修了表面问题
    影响范围评估不足
    测试回归场景不完整
    需求理解不一致
    环境或数据导致误判
    
如果一个模块反复重开，说明不能只看“修了多少”，还要看“修得稳不稳”。
### 逃逸缺陷：判断测试策略有没有漏
线上发现的问题，叫逃逸缺陷。
可以看两个指标：
    
    逃逸缺陷数
    逃逸率 = 线上逃逸缺陷数 / 测试阶段发现缺陷总数
    
但逃逸缺陷最重要的不是数量，而是级别和原因。

线上问题| 漏测原因  
---|---  
支付回调重复处理| 幂等场景漏测  
优惠券退款后未返还| 状态流转漏测  
权限接口可越权访问| 只测了前端按钮权限  
大数据量查询超时| 性能场景未覆盖  

逃逸缺陷不是为了追责，而是反推测试策略：
    
    哪些场景没覆盖？
    哪些风险没识别？
    哪些自动化需要补上？

### 哪些数据最容易误导人
**1\. 用例通过率高**
可能只是用例设计太浅，只测了正常流程。
**2\. 缺陷数量少**
可能是测试没测透、时间太短、数据不足。
**3\. 自动化通过率高**
只能说明已有脚本覆盖的部分稳定，没覆盖的风险仍然存在。
**4\. 修复率高**
如果剩下的是支付、权限、库存问题，依然不能轻易上线。
所有指标都可能被“美化”。所以质量评估不能只看结果指标，还要看过程指标和风险指标。
### 总结
项目质量评定不要只看冒烟通过率，也不要只看缺陷密度。
    
    冒烟通过率：版本能不能继续测
    缺陷密度：问题集中在哪里
    缺陷级别：有没有上线阻断风险
    修复趋势：质量是否收敛
    重开率：修复质量是否稳定
    逃逸缺陷：测试策略是否漏风险
    
最后真正要回答的是：
    
    核心链路是否稳定？
    严重风险是否清零？
    遗留问题是否可接受？
    上线后最可能炸在哪里？
    
一句话：**项目质量不是看一个数字漂不漂亮，而是看风险有没有被识别、被修复、被接受。**
