# [AI测试] 接口测试数据怎么准备？让 AI 帮你覆盖正常、异常和边界场景

> 原文: https://blog.csdn.net/weixin_42390585/article/details/161315394

> 原创内容，未获授权禁止转载、转发、抄袭。

做接口测试时，很多人会让 AI 直接“生成几组测试数据”。
这个做法不太稳。因为接口数据不是几段 JSON，而是字段规则、业务状态和预期结果的组合。
我更推荐的方式是：**先提规则，再拆场景，再造数据，最后用脚本校验和执行。**

### 先别急着造数据
接口测试数据一般不是孤立字段，而是业务规则的组合。
比如订单创建接口：
    
    POST /api/orders/create
    
字段大概是：
    
    {
      "userId": 10001,
      "skuId": 1001,
      "count": 1,
      "couponCode": "TEST20",
      "payType": "WECHAT"
    }
    
如果直接让 AI 生成数据，它可能只会给你几组“正常 JSON”。但接口真正容易出问题的地方，通常在这些规则里：
    
    userId 是否存在
    skuId 是否上架
    count 是否超限
    couponCode 是否可用、过期、已使用
    payType 是否在枚举范围内
    库存是否足够
    金额是否重新计算
    
所以第一步不是造数据，而是先把数据约束拆出来。
我一般会拆成三层：

层级| 要看什么| 例子  
---|---|---  
字段约束| 类型、必填、长度、枚举、范围| `count` 必填，范围 1~99  
业务状态| 数据是否存在、是否可用、状态是否正确| 商品已上架、优惠券未过期  
结果校验| 接口返回、数据库、库存、金额| 订单创建、库存扣减、金额重算  

只让 AI 看字段，数据很容易“格式正确、业务无效”。
### 第一个提示词：先提约束
可以这样问：
    
    你是测试开发，请根据下面的接口信息提取测试数据约束。
    
    接口：POST /api/orders/create
    
    字段：
    - userId: number，必填，必须是已注册用户
    - skuId: number，必填，必须是已上架商品
    - count: number，必填，范围 1~99
    - couponCode: string，可选，必须是当前用户可用优惠券
    - payType: enum[WECHAT, ALIPAY]
    
    要求：
    1. 不要直接生成测试数据
    2. 先输出字段约束表
    3. 再输出正常、异常、边界、组合场景
    4. 每个场景说明命中的规则和预期结果
    
这个提示词的重点是：**先让 AI 理解规则，不要一上来就编数据。**

### 第二步：生成测试数据矩阵
规则拆完后，再让 AI 生成数据矩阵：
    
    基于上面的字段约束，生成订单创建接口测试数据矩阵。
    
    要求：
    1. 覆盖正常、异常、边界、组合场景
    2. 每条数据包含：场景、请求数据、预期结果、校验点
    3. 数据要尽量可执行，不要只写描述
    4. 用 Markdown 表格输出
    
示例结果可以长这样：  
![在这里插入图片描述](/cdn/161315394/948491cf1b4f4d468c3b59f3aab19b12.png)
这类表格比单纯 JSON 更有用，因为它把“为什么要造这条数据”说清楚了。后面评审测试用例时，也能一眼看出有没有漏掉关键场景。
### 第三步：把数据变成可执行用例
拿到矩阵后，再让 AI 输出结构化数据，比如：
    
    [
      {
        "name": "正常-无优惠券下单",
        "data": {
          "userId": 10001,
          "skuId": 1001,
          "count": 1,
          "couponCode": null,
          "payType": "WECHAT"
        },
        "expect": {
          "success": true,
          "check": ["订单创建成功", "金额按原价计算", "库存扣减"]
        }
      },
      {
        "name": "异常-count=0",
        "data": {
          "userId": 10001,
          "skuId": 1001,
          "count": 0,
          "couponCode": null,
          "payType": "WECHAT"
        },
        "expect": {
          "success": false,
          "error": "购买数量不合法"
        }
      }
    ]
    
然后接口自动化里直接遍历执行。
以 Playwright API 测试为例：
    
    import { expect, test } from '@playwright/test';
    import cases from './order-create.cases.json';
    
    for (const item of cases) {
      test(item.name, async ({ request }) => {
        const res = await request.post('/api/orders/create', {
          data: item.data
        });
    
        const body = await res.json();
    
        if (item.expect.success) {
          expect(res.status()).toBe(200);
          expect(body.code).toBe(0);
          expect(body.data.orderId).toBeTruthy();
        } else {
          expect(body.code).not.toBe(0);
          expect(body.message).toContain(item.expect.error);
        }
      });
    }
    
这里有个细节：不要只断言 `code` 和 `message`。
订单类接口至少还要看：
    
    订单是否落库
    金额是否正确
    库存是否变化
    优惠券状态是否变化
    失败场景是否没有产生脏数据
    
这样生成的数据才不是文档里的摆设，而是真能跑进自动化里的测试资产。
### 数据不能只覆盖“能成功”
接口测试数据至少要覆盖这几类：

类型| 例子| 重点  
---|---|---  
正常数据| 普通下单、优惠券下单| 主流程能跑通  
异常数据| 商品不存在、用户不存在| 错误码和提示  
边界数据| count=1、count=99、count=100| 边界规则  
组合数据| 过期优惠券 + 正常商品| 业务规则冲突  
状态数据| 商品下架、库存不足| 状态流转  

很多接口问题不是单字段导致的，而是组合条件导致的。
比如：
    
    优惠券本身存在，但已过期
    商品存在，但已下架
    库存足够，但用户超过限购
    订单创建成功，但金额没有重新计算
    
这些场景，如果只靠随机生成数据，很容易漏掉。
### AI 造数最容易踩的坑
我实际用下来，主要有几个坑。
**1\. 只生成正常数据**
AI 默认很喜欢给“看起来合理”的数据，但测试不能只看合理路径。
提示词里要明确写：
    
    正常、异常、边界、组合场景都要覆盖。
    
**2\. 枚举值乱编**
比如实际只支持：
    
    WECHAT, ALIPAY
    
AI 可能给你生成：
    
    CREDIT_CARD
    BALANCE
    
所以枚举字段必须在提示词里写死。
**3\. 只看字段，不看状态**
接口能不能成功，不只看参数格式，还看业务状态。
比如：
    
    skuId=1001
    
这个商品可能是上架、下架、库存不足、限购中。状态不同，预期结果完全不同。
**4\. ID 看起来真实，但实际不可用**
AI 很容易生成这种数据：
    
    {
      "userId": 10001,
      "skuId": 1001
    }
    
问题是：这些 ID 在你的测试环境里不一定存在。
更稳的做法是用接口或 SQL 先准备数据：
    
    createTestUser()
    createTestSku()
    createTestCoupon()
    
再把真实返回的 ID 填进请求里。
**5\. 数据不可重复执行**
优惠券、订单号、手机号这类数据，很容易被使用一次后失效。
建议给自动化数据加标记：
    
    AUTO_TEST_20260522_001
    
执行后按标记清理，或者每次执行重新创建。
### 我更推荐的落地方式
接口测试数据可以按这个顺序做：
    
    1. 从接口文档提取字段约束
    2. 从需求规则提取业务约束
    3. 让 AI 生成测试数据矩阵
    4. 人工 review 场景是否漏掉
    5. 让 AI 输出 JSON / CSV / SQL
    6. 自动化脚本执行并断言
    7. 执行后清理测试数据
    
实际落地时，我会加一个简单的检查清单：
    
    字段是否合法
    枚举是否真实存在
    ID 是否来自测试环境
    状态是否符合场景
    失败场景是否会产生脏数据
    执行后是否可清理
    
AI 适合帮我们扩展思路、补充边界、批量生成数据。但最终是否能跑、是否符合业务规则，还是要靠测试人员把关。
### 总结
AI 可以帮接口测试造数据，但不要让它“自由发挥”。
比较靠谱的方式是：
    
    先提约束
    再拆场景
    再造数据
    最后校验
    
一句话：**AI 不是测试数据的来源，规则才是。AI 只是帮我们把规则更快地变成可执行数据。**
