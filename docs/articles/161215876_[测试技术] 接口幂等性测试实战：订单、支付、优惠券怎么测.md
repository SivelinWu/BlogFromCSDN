# [测试技术] 接口幂等性测试实战：订单、支付、优惠券怎么测

> 原文: https://blog.csdn.net/weixin_42390585/article/details/161215876

> 原创内容，未获授权禁止转载、转发、抄袭。

接口幂等性是测试里很容易被低估的一块。
很多接口点一次没问题，但遇到重复点击、网络重试、前端超时重发、MQ 重复消费，就可能出事故。
典型问题包括：
  * 重复下单
  * 重复扣款
  * 重复发券
  * 库存重复扣减
  * 优惠券重复核销
  * 订单状态被错误覆盖


这篇不展开讲理论，直接说怎么测。
* * *
### 一、幂等性到底测什么
简单说：
同一个业务请求，执行一次和执行多次，最终业务结果应该一致。
注意重点是 **最终业务结果一致** ，不是每次接口返回完全一样。
比如创建订单接口。
第一次返回：
    
    {
      "code": "SUCCESS",
      "orderId": "O202605190001"
    }
    
第二次重复请求，返回同一个订单可以：
    
    {
      "code": "SUCCESS",
      "orderId": "O202605190001"
    }
    
返回已处理也可以：
    
    {
      "code": "ORDER_ALREADY_CREATED",
      "orderId": "O202605190001"
    }
    
但如果第二次又生成新订单，就有问题：
    
    {
      "code": "SUCCESS",
      "orderId": "O202605190002"
    }
    
一句话：
不怕重复请求，怕重复生效。
* * *
### 二、哪些接口要重点测
优先关注会改变数据或资产的接口。

类型| 示例| 风险  
---|---|---  
创建类| 创建订单、创建工单| 重复创建  
支付类| 支付确认、退款申请| 重复扣款、重复退款  
资产类| 领券、发积分、发余额| 重复发放  
扣减类| 扣库存、核销券| 重复扣减  
状态流转| 取消订单、确认收货| 状态错乱  
回调类| 支付回调、物流回调| 重复消费  

查询类接口风险低，但如果查询接口里夹带写操作，也要测。
* * *
### 三、通用测试方法
幂等性测试的核心操作就四类：

方法| 怎么测| 看什么  
---|---|---  
连续重复| 同一请求连续发 2、5、10 次| 是否重复生成、重复扣减  
并发重复| 同一请求同时并发发出| 是否只有一次成功  
超时重试| 第一次请求超时后再次请求| 是否重复执行业务  
消息重复| 同一回调或 MQ 消息重复投递| 是否重复消费  

注意：  
这里说的“同一请求”，不是参数看起来差不多，而是业务上代表同一次操作。
常见幂等标识有：
    
    requestId
    orderNo
    payNo
    transactionId
    couponUserId
    messageId
    
没有幂等标识的接口，要重点关注。
幂等号也不能随便设计。  
比如订单场景不能简单用 `userId + productId` 当幂等标识，否则用户正常再次购买同一商品，可能会被误判为重复请求。
* * *
### 四、实操：怎么发重复请求
如果只是手工验证，用 Postman、Apifox、JMeter 都可以。
#### 1\. 连续重复请求
同一个请求体，连续发送多次。
    
    {
      "userId": "U10001",
      "productId": "P20001",
      "quantity": 1,
      "requestId": "REQ202605190001"
    }
    
重点是：`requestId` 不变。
#### 2\. 并发重复请求
用 JMeter 或脚本同时发请求。
简单思路：
    
    线程数：10
    循环次数：1
    请求参数：完全一致
    requestId：固定不变
    
预期：
    
    只成功一次，或者都返回同一业务结果。
    
不应该出现：
    
    10 个请求生成 10 条订单。
    
#### 3\. 超时重试
可以让服务端处理较慢，或者模拟客户端超时后重发。
关注点不是第一次有没有返回，而是：
    
    第二次请求是否复用了第一次处理结果。
    
#### 4\. 回调重复
支付回调、物流回调、MQ 消费，要用同一个消息号重复投递。
重点看：
    
    同一 transactionId / messageId 是否只处理一次。

### 五、订单接口怎么测
以创建订单为例：
    
    POST /api/order/create
    
请求示例：
    
    {
      "userId": "U10001",
      "productId": "P20001",
      "quantity": 1,
      "requestId": "REQ202605190001"
    }
    
测试重点：

场景| 操作| 预期  
---|---|---  
正常创建| 发送一次请求| 创建成功  
重复创建| 同一 `requestId` 请求多次| 只生成一个订单  
并发创建| 同一 `requestId` 并发请求| 只成功一次或返回同一订单  
不同请求号| 更换 `requestId` 再请求| 按业务规则创建新订单  
超时重试| 第一次超时后重发| 不重复创建  
库存校验| 重复请求后查库存| 库存只扣一次  

重点查这些数据：
    
    订单表：是否只有一条订单
    库存：是否只扣减一次
    金额：是否一致
    用户订单列表：是否只展示一笔
    
常见坑：
  * 只做前端防重复点击，后端没防
  * 幂等号设计不合理，影响用户再次购买
  * 幂等号过期太快，重试时已失效
  * 并发下数据库唯一约束异常直接抛给用户


* * *
### 六、支付接口怎么测
支付类接口风险最高。
常见接口：
    
    POST /api/pay/confirm
    POST /api/pay/callback
    POST /api/refund/apply
    
测试重点：

场景| 操作| 预期  
---|---|---  
支付确认重复| 同一支付单重复确认| 只扣款一次  
支付回调重复| 同一回调重复发送| 订单只更新一次  
回调乱序| 先成功后失败| 成功状态不能被失败覆盖  
退款重复申请| 同一退款单重复请求| 只生成一笔退款  
权益发放| 支付成功重复回调| 积分、会员、券只发一次  

重点查这些数据：
    
    支付流水：是否只有一条成功流水
    订单状态：是否正确
    账户余额：是否只扣一次
    退款记录：是否只生成一次
    权益发放：是否只发一次
    回调日志：是否能识别重复通知
    
支付回调建议重点测：
    
    同一个 transactionId 重复回调 5 次。
    
可以接受：
    
    {
      "code": "SUCCESS",
      "message": "already processed"
    }
    
不能接受：
    
    重复扣款、重复发权益、订单状态被失败回调覆盖。
    
这里要特别注意状态优先级。  
支付成功通常是终态，后续失败、关闭等低优先级回调，不应该覆盖成功状态。

### 七、优惠券接口怎么测
优惠券是幂等问题高发区。
常见接口：
    
    POST /api/coupon/receive
    POST /api/coupon/use
    POST /api/coupon/refund
    
#### 1\. 领券接口

场景| 操作| 预期  
---|---|---  
正常领券| 用户领取一次| 领取成功  
重复领券| 同一用户重复领取| 只发一张券  
并发领券| 同一用户多端同时点击| 只成功一次  
库存扣减| 重复请求领券| 库存只扣一次  
库存不足| 库存为 0 时请求| 不发券  

重点查：
    
    用户券包：是否只有一张券
    券库存：是否只扣减一次
    领券记录：是否只有一条
    接口返回：是否明确提示已领取
    
#### 2\. 用券接口

场景| 操作| 预期  
---|---|---  
正常核销| 下单时使用优惠券| 核销成功  
重复核销| 同一张券重复使用| 只能成功一次  
并发核销| 两个订单同时用同一张券| 只能一个订单成功  
下单失败| 订单创建失败| 优惠券按业务规则恢复  
取消订单| 订单取消| 按业务规则退券或保持已使用  

重点查：
    
    优惠券状态：未使用 → 已使用
    订单优惠金额：是否只抵扣一次
    用户券包状态：是否正确
    取消或失败后：是否按规则处理，不能重复退券
    
优惠券最容易漏的是：
    
    同一用户、同一张券、两个订单同时提交。
    
如果没控制好，两个订单都可能享受优惠。
* * *
### 八、判断是否幂等，不要只看返回
接口返回 `SUCCESS` 不代表真的没问题。
建议从这几层查：

校验层| 看什么  
---|---  
接口返回| 返回码、业务码、幂等提示  
数据库| 订单、流水、库存、券包记录  
用户页面| 列表是否重复展示  
日志| 是否识别重复请求  
消息| MQ 是否重复消费  
资产| 是否重复扣减或发放  

幂等性测试真正要看的不是“接口有没有报错”，而是：
    
    业务是否只生效一次。

### 九、用例模板
可以直接按这个模板写。

字段| 示例  
---|---  
用例名称| 创建订单接口重复请求幂等性测试  
前置条件| 用户有库存充足商品，生成固定 requestId  
请求接口| `POST /api/order/create`  
操作步骤| 使用同一 requestId 连续请求 5 次  
预期结果| 只生成一笔订单，库存只扣一次  
数据校验| 订单表、库存表、用户订单列表  
优先级| P0  

示例：
    
    用例名称：同一用户重复领取优惠券
    
    前置条件：
    1. 用户 U10001 未领取该券
    2. 优惠券 C10001 库存为 10
    
    步骤：
    1. 使用用户 U10001 调用领券接口
    2. 使用相同用户再次调用领券接口 5 次
    3. 查询用户券包
    4. 查询优惠券库存
    5. 查询领券记录
    
    预期：
    1. 用户券包中只有一张 C10001
    2. 优惠券库存只减少 1
    3. 领券记录只有一条
    4. 重复请求返回“已领取”或同等业务提示

### 十、常见问题

问题| 说明  
---|---  
只做前端防重复| 后端接口仍可重复调用  
幂等号不唯一| 不同业务请求被错误合并  
幂等号过期太快| 网络重试时失效  
并发控制不足| 多线程下仍重复成功  
只看返回不查数据| 表面成功，数据已错  
回调乱序未处理| 成功状态被失败覆盖  
重复消息未去重| MQ 重复消费导致重复处理  

* * *
### 总结
接口幂等性测试不用想复杂。
核心方法是：
**一个业务请求，重复发、并发发、超时后重发、消息重复发。**
核心判断是：
**业务结果是否只生效一次。**
三个高风险场景记住这几句：
**订单看是否重复创建和重复扣库存。
支付看是否重复扣款、重复退款和状态覆盖。
优惠券看是否重复发券、重复核销和库存券包一致性。**
